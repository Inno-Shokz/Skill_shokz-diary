# Shokz Diary 企业级日报 Skill 设计

## 背景

团队正在试行 AI 辅助工作复盘机制，原因是近期暴露出组织节奏松散、成员推进节奏不一致、导师与学员沟通成本高等问题。日报 skill 不是简单的写作助手，而是团队管理、定期复盘和常见问题沉淀的入口。

核心目的：

1. 加强团队管理：建立固定检查与监督节奏，减少松散推进。
2. 推动定期复盘：让成员明确自身目标，持续回看工作质量。
3. 沉淀常见问题：把复盘中反复出现的问题沉淀为可复用团队文档。

## 目标

该 skill 的首要目标是帮助学员产出一份企业级、可提交、导师可直接批阅的日报。日报应降低导师阅读和追问成本，同时不能因 AI 润色而隐藏学员原始表达中的问题。

成功标准：

- 导师能在 2 分钟内读完正文并理解当天推进情况、风险和需要确认的事项。
- 学员原始表达完整保留在附录中，便于追溯和复盘。
- Agent 不把推断包装成事实，不为了让日报好看而掩盖信息缺口。
- 每次使用后，系统能基于确认过的信息更新学员动态画像和导航索引。

## 现有框架

当前项目已有插件骨架，不需要推倒重建：

- `.codex-plugin/plugin.json`：Codex 插件声明。
- `skills/diary-generation/SKILL.md`：日报生成规则。
- `commands/shokz-init.md`：初始化学员 profile。
- `commands/shokz-diary.md`：每日复盘主流程。
- `commands/shokz-calibrate.md`：校准当天日报。
- `.shokz/profile.json`：学员稳定配置。
- `.shokz/reports/`：日报输出目录。

升级方向是在现有框架上改造交互规则、日报模板、质量标准和连续记忆机制。

## 核心角色定位

Agent 的角色不是“代写日报的人”，而是：

1. 保真记录员：完整保留学生原话和交互过程。
2. 追问教练：针对模糊、空泛、高风险输入进行一次必要追问。
3. 结构化中间件：把学生输入转换为导师可读、可批阅、可跟进的企业级日报。
4. 记忆维护者：在用户确认后更新动态学员画像、索引和未关闭事项。

## 日报最终形态

采用“日报正文 + 附录原始对话”的结构。

### 正文

正文面向导师，应尽量控制在一页内，包含：

1. 基本信息：姓名、导师、日期、实习目标、当日主题。
2. 今日推进：按事项列出实际推进内容、产出、状态和来源。
3. 关键结论：列出当天已经形成的结论；没有结论时写“尚未形成明确结论”。
4. 卡点与风险：区分执行阻塞、认知卡点、信息缺口和决策风险。
5. 明日计划：列出 1-3 个可执行事项，每项含做法和完成标志。
6. 需导师确认：将模糊问题改写为导师可回答的问题。
7. 管理视角补充：标注本日暴露的问题类型，如目标不清、信息未结构化、决策依据不足、汇报表达不清。
8. 问题沉淀候选：若出现可复用团队问题，生成建议沉淀条目；否则写“暂无”。
9. 导师反馈区：保留导师批注空间。

### 附录

附录完整保留：

1. Agent 首轮问题。
2. 学员原始回答。
3. Agent 事实清单确认。
4. Agent 追问。
5. 学员补充。
6. 画像更新建议和用户确认记录。

附录不得摘要替代原话，不得为了可读性改写学员表达。

## 信息来源标记

正文中的关键信息应尽量标注来源，防止信息失真。

可用来源类型：

- 学生原话：来自学员初始回答。
- 追问补充：来自 Agent 追问后的补充回答。
- Agent 推断：Agent 基于上下文推导出的判断。
- 待确认：证据不足，不能作为事实写死。

规则：

- 事实性内容优先来自“学生原话”或“追问补充”。
- “Agent 推断”只能用于风险、问题类型和建议，不得伪装成事实。
- 不确定信息必须标注“待确认”，不能脑补。

## 交互流程

### 1. 读取上下文

每次日报开始前读取：

- `.shokz/profile.json`
- `.shokz/learner-profile.md`（若存在）
- `.shokz/index.md`（若存在）
- 最近 3-5 篇日报（若存在）
- 最近未关闭的待跟进事项和导师反馈（若存在）

### 2. 收集原始输入

Agent 让学员自由回答当天情况，问题应覆盖：

- 今天做了什么。
- 形成了什么结论或产出。
- 哪里卡住了或仍没想清楚。
- 明天准备怎么推进。
- 需要导师确认什么。

Agent 必须完整保留学员原话。

### 3. 事实清单确认

Agent 根据原始回答抽取事实清单，并让学员确认是否漏掉或歪曲。

事实清单包括：

- 推进事项。
- 具体产出。
- 已形成结论。
- 使用的数据、观察或证据。
- 未解决问题。
- 明日计划。

未经确认的事实不得直接进入最终正文。

### 4. 高风险回答追问

若学员出现以下回答，Agent 必须追问一次，不能直接进入生成：

- “无卡点”
- “都很顺利”
- “没问题”
- “无”
- 只有流水账，没有结论、证据或下一步

推荐追问：

> 你可以没有执行阻塞，但今天至少需要确认一个“认知卡点”或“待验证结论”。请从下面选一个回答：
>
> 1. 今天哪个问题还没想清楚？
> 2. 哪个结论还缺证据？
> 3. 哪个决策还需要导师确认？
> 4. 如果都没有，请说明今天形成了什么明确结论，以及证据是什么。

追问不应制造问题，而是防止空泛表达掩盖思考缺口。

### 5. 生成日报正文

只有在事实清单确认和必要追问完成后，Agent 才能生成企业级日报正文。

生成规则：

- 用专业、直接、可批阅的语言。
- 不使用鼓励式空话。
- 不把学生原话压缩到丢失信息。
- 不确定内容写“待确认”。
- 明日计划必须可执行、可验证。

### 6. 生成附录

将完整交互记录写入附录。附录承担证据链功能，不追求文风优化。

### 7. 生成更新建议

日报生成后，Agent 生成但不自动写入以下建议：

- 学员画像更新建议。
- 导航索引更新建议。
- 未关闭事项更新建议。
- 可沉淀团队问题建议。

学员确认后，Agent 再写入对应文件。

## 连续记忆设计

### `.shokz/profile.json`

存放稳定配置：

- 姓名
- 导师
- 核心实习目标
- 入职日期
- 输出目录
- 创建时间

稳定配置不应混入动态评价，避免长期画像污染基础信息。

### `.shokz/learner-profile.md`

存放动态学员画像，建议结构：

- 当前项目与职责
- 当前阶段目标
- 长期 gap
- 表达习惯与汇报偏好
- 常见卡点
- 导师关注点
- 已确认成长变化
- 已过期或被修正的观察

每条画像观察必须包含：

- 观察内容
- 证据来源
- 首次出现日期
- 最近验证日期
- 状态：观察中 / 已确认 / 已修正 / 已过期

### `.shokz/index.md`

作为导航索引，建议包含：

- 按日期排列的日报链接
- 按项目排列的日报链接
- 按问题类型排列的问题索引
- 导师反馈索引
- 未关闭事项
- 团队文档沉淀候选

### 更新原则

- 默认只生成更新建议，不自动写入。
- 学员确认后才能写入。
- 学员可以要求覆盖、删除、降级、标记为过期。
- Agent 不得永久标签化学员。
- 一次表达错误不能变成长期定性。

## 校准流程

`shokz-calibrate` 应支持两类校准：

1. 日报正文校准：修改正文中的措辞、结构、事实或建议。
2. 画像校准：修改、删除、降级或标记过期的画像观察。

校准时必须保留原始对话附录，不得反向改写历史原话。

## 质量红线

- 不压缩学生原话为摘要；原始对话必须完整进入附录。
- 不把 Agent 推断写成事实。
- 不确定处标“待确认”。
- 不接受空泛输入直接生成。
- 不为了让日报好看而隐藏学生表达问题。
- 不永久标签化学员。
- 不让画像未经确认自动沉淀。
- 不把导师反馈吞掉；反馈应进入下一次日报上下文和索引。

## 非目标

本次升级不追求：

- 搭建数据库或后台服务。
- 构建 UI。
- 自动同步到外部协作平台。
- 替导师直接评价绩效。
- 用 AI 完全替代导师沟通。

## 验收标准

使用 2026-07-05 失败案例验证时，新流程应满足：

- 学员原始表达不会被压缩为一句话。
- “没有卡点”会触发一次认知卡点追问。
- 正文能指出华强北调研缺少结构化信息。
- 正文能把 C 端定位问题拆成需导师确认或明日补充事项。
- 附录能追溯学生原话和追问过程。
- 日报末尾能生成画像更新建议，例如“表达清晰度需要观察”，但不会未经确认写入长期画像。

## 实施建议

优先改造顺序：

1. 更新 `commands/shokz-diary.md`，加入上下文读取、事实确认、高风险追问、附录和更新建议流程。
2. 更新 `skills/diary-generation/SKILL.md`，替换 A/B/C/D 结构为企业级日报正文结构。
3. 更新 `skills/diary-generation/references/复盘模板.md`，同步新模板。
4. 新增或更新 `.shokz/learner-profile.md` 和 `.shokz/index.md` 的模板说明。
5. 更新 `commands/shokz-calibrate.md`，支持日报校准与画像校准。

实施前应先基于本设计写具体实现计划。
