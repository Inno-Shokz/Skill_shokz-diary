# 日报生成 Skill

本 Skill 定义如何把学员输入转化为企业级可提交日报。它只负责结构化和表达升级，不负责改写原始对话。原始对话必须由主流程完整放入附录。

## 输入

使用以下信息生成日报：

- 学员稳定配置：`.shokz/profile.json`
- 动态学员画像：`.shokz/learner-profile.md`（若存在）
- 导航索引与未关闭事项：`.shokz/index.md`（若存在）
- 最近 3-5 篇日报与导师反馈（若存在）
- 学员当日原始回答全文
- 事实清单确认记录
- Agent 追问与学员补充

## 输出目标

生成一份导师可直接批阅的日报正文，并给出画像/索引更新建议。

日报正文应满足：

- 导师 2 分钟内能读完并理解重点。
- 能回答“今天推进了什么、有什么结论、风险在哪里、明天怎么做、需要导师确认什么”。
- 保留信息来源，避免把推断伪装成事实。
- 暴露信息缺口，不用漂亮语言遮住学员汇报问题。

## 来源标记

正文中的关键事实和判断必须标注来源。

来源类型固定为：

- `学生原话`：来自学员初始回答。
- `追问补充`：来自高风险追问后的补充。
- `Agent 推断`：基于上下文做出的风险、问题类型或建议判断。
- `待确认`：证据不足，不能写成事实。

规则：

- 今日推进、产出、数据、结论等事实性内容优先使用 `学生原话` 或 `追问补充`。
- `Agent 推断` 只能用于风险提示、管理视角、行动建议和沉淀候选。
- 缺少证据时写 `待确认`，不要脑补。
- 不要把“看起来合理”的内容补成学员已经完成的事实。

## 日报正文结构

严格生成以下区块。

### 1. 基本信息

输出表格：

| 字段 | 内容 |
|------|------|
| 姓名 | |
| 导师 | |
| 日期 | |
| 用时 | |
| 实习目标 | |
| 当日主题 | |

规则：

- 当日主题从事实清单中提炼，最多 20 个字。
- 若当日主题无法从输入确认，写 `待确认`。

### 2. 今日推进

输出表格：

| 事项 | 推进内容 | 产出/证据 | 状态 | 来源 |
|------|----------|-----------|------|------|

规则：

- 按事项拆分，不写流水账。
- 每个事项必须有 `产出/证据`；没有则写 `待补充`。
- 状态使用：`已完成` / `推进中` / `待确认` / `有风险`。
- 最多 5 行，优先保留导师需要判断的信息。

### 3. 关键结论

输出表格：

| 结论 | 依据 | 来源 | 可信度 |
|------|------|------|--------|

规则：

- 只写已经形成或可被输入支撑的结论。
- 没有结论时写：`尚未形成明确结论`，并说明缺少什么证据。
- 可信度使用：`高` / `中` / `低` / `待确认`。
- 不要把行动建议写成结论。

### 4. 卡点与风险

输出表格：

| 类型 | 描述 | 影响 | 建议处理 | 来源 |
|------|------|------|----------|------|

类型固定为：

- `执行阻塞`
- `认知卡点`
- `信息缺口`
- `决策风险`
- `表达/汇报问题`

规则：

- 如果学员说“无卡点”，但输入显示缺少结论、证据、标准或下一步，要列为 `认知卡点` 或 `信息缺口`，来源标注 `Agent 推断`。
- 不羞辱学员，不做人格评价，只描述工作风险。
- 最多 4 行，按影响程度排序。

### 5. 明日计划

输出表格：

| 行动 | 怎么做 | 完成标志 | 优先级 |
|------|--------|----------|--------|

规则：

- 输出 1-3 项。
- `怎么做` 必须是步骤级行动。
- `完成标志` 必须可验证。
- 优先级使用：`高` / `中` / `低`。
- 明日计划必须回应今日风险和导师反馈，不能泛泛写“继续推进”。

### 6. 需导师确认

输出：

```markdown
**需导师确认：**
1. ...

**需导师提供输入：**
1. ...
```

规则：

- 将学员模糊问题改写成导师可回答的问题。
- 优先写成 Yes/No、选择题或具体输入请求。
- 如果学员写“无”，但存在定位、标准、优先级、资源依赖等风险，要提出“建议确认”的问题，并标注来源为 `Agent 推断`。
- 如果确实无需导师确认，写 `本日暂无需导师确认事项`。

### 7. 管理视角补充

输出表格：

| 问题类型 | 观察 | 建议 |
|----------|------|------|

问题类型可选：

- `目标不清`
- `信息未结构化`
- `决策依据不足`
- `汇报表达不清`
- `行动闭环不足`
- `导师输入依赖`
- `无明显管理问题`

规则：

- 这是团队管理视角，不是给学员贴标签。
- 每条观察必须基于当天输入或历史反馈。
- 最多 3 行。

### 8. 问题沉淀候选

输出表格：

| 候选主题 | 为什么值得沉淀 | 建议文档方向 |
|----------|----------------|--------------|

规则：

- 只沉淀可复用问题，如调研前准备清单、产品定位讨论框架、日报表达标准。
- 如果没有可复用问题，写 `暂无`。
- 不要把个人情绪或一次性细节沉淀成团队文档。

### 9. 导师反馈区

输出表格：

| 反馈项 | 导师意见 |
|--------|----------|
| 对今日推进的补充 | |
| 对关键结论的判断 | |
| 对卡点/风险的建议 | |
| 对明日计划的调整 | |
| 其他 | |

## 附录 B：更新建议

日报正文后必须生成更新建议，但默认不写入长期文件。

### 学员画像更新建议

输出表格：

| 操作 | 观察内容 | 证据来源 | 建议状态 |
|------|----------|----------|----------|

操作使用：

- `新增`
- `更新`
- `关闭`
- `降级`
- `标记过期`
- `暂无`

建议状态使用：

- `观察中`
- `已确认`
- `已修正`
- `已过期`

规则：

- 一次表现只能建议为 `观察中`，不要直接写成稳定结论。
- 必须写证据来源，如 `2026-07-05 日报原始对话`、`导师反馈`。
- 如果没有足够证据，写 `暂无`。

### 导航索引更新建议

输出表格：

| 索引项 | 建议写入内容 |
|--------|--------------|

索引项包括：

- 日期日报
- 项目索引
- 问题类型索引
- 导师反馈索引
- 未关闭事项
- 团队文档沉淀候选

## 质量标准

- 直接、专业、可批阅。
- 不使用空洞鼓励语。
- 不用模板化套话掩盖信息不足。
- 不压缩或改写原始对话；原始对话由主流程放入附录。
- 不生成未经证据支持的事实。
- 所有建议必须能追溯到输入、追问、历史画像或导师反馈。

## 禁止行为

- 不要输出旧版 A/B/C/D 主结构。
- 不要把“无卡点”直接解释为没有问题。
- 不要替学员虚构结论、数据、产出或导师意图。
- 不要把一次表达问题永久标签化。
- 不要自动写入 `.shokz/learner-profile.md` 或 `.shokz/index.md`；只能给建议，等待主流程获得确认。
